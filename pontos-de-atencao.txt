Dei uma varrida no projeto inteiro. Minha leitura: a estrutura já está indo para um padrão bom, mas ainda tem umas “portas laterais” que deixam cada pessoa programar de um jeito.

**Principais problemas encontrados**

1. **Não existe enforcement hoje**
   
   O [package.json](/home/aalps/Documents/TCC/StudoCore-App/package.json:5) só tem:

```json
"start", "android", "ios", "web"
```

Não tem `lint`, `typecheck`, `format`, `test`, `check`. Então mesmo com `strict: true` no [tsconfig.json](/home/aalps/Documents/TCC/StudoCore-App/tsconfig.json:4), ninguém é obrigado a validar nada antes de subir código.

Eu adicionaria:

```json
"lint": "eslint .",
"typecheck": "tsc --noEmit",
"check": "npm run lint && npm run typecheck"
```

2. **Supabase ainda está vazando para fora de `services`**

Exemplos claros:

- [hooks/useMyGroups.ts](/home/aalps/Documents/TCC/StudoCore-App/hooks/useMyGroups.ts:3) importa `@/lib/supabase`
- [hooks/useMyGroups.ts](/home/aalps/Documents/TCC/StudoCore-App/hooks/useMyGroups.ts:23) chama `supabase.auth.getUser()`
- [components/ui/ImagePickerAvatar.tsx](/home/aalps/Documents/TCC/StudoCore-App/components/ui/ImagePickerAvatar.tsx:5) importa Supabase, embora nem pareça usar diretamente
- Também aparece em `hooks/useAuth.ts`, `hooks/useAuthState.ts`, `hooks/useArchives.ts`, `hooks/useOnlineUsers.ts`, `hooks/usePublicGroups.ts`, `hooks/useGroupMembers.ts`, `hooks/useMemberGroupStatus.ts`

Sugestão de regra: **somente `services/**` e `lib/supabase.ts` podem importar `@/lib/supabase`**.

4. **Mistura de imports com alias e relativos**

Tem arquivos usando `@/services/...`, outros usando `../services/...`.

Exemplos:

9. **Services sem contrato consistente**

Alguns services retornam diretamente resposta do Supabase:

```ts
return await supabase...
```

Outros fazem `try/catch`, logam erro e retornam `[]`, `null`, `0` ou `undefined`.

Isso dificulta a tela saber o que esperar.

Eu padronizaria uma dessas abordagens:

```ts
type Resultado<T> =
  | { sucesso: true; dados: T }
  | { sucesso: false; erro: string };
```

ou então services sempre lançam erro, e hooks/telas tratam.

10. **Hooks fazendo papel de service**

[hooks/useMyGroups.ts](/home/aalps/Documents/TCC/StudoCore-App/hooks/useMyGroups.ts:23) busca usuário, consulta tabela, faz cache e transforma dados. Isso é muita regra de backend dentro do hook.

Eu deixaria:

```txt
services/groups.ts
  buscarMeusGrupos()

hooks/useMyGroups.ts
  loading, refresh, cache, estado da UI
```

**Regras que eu colocaria agora**

Prioridade alta:

```txt
- app/** não importa @/lib/supabase
- components/** não importa @/lib/supabase
- hooks/** não importa @/lib/supabase, exceto talvez hooks de auth enquanto refatora
- proibido any
- proibido console.log em produção
- imports internos via @/
- npm run check obrigatório antes do merge
```

Prioridade média:

```txt
- mocks não podem ser importados por telas finais
- services devem ter retorno tipado explícito
- nomes de funções de domínio em português
- arquivos SQL devem ficar só em migrations ou docs/sql
- .idea fora do Git
```

Não consegui rodar o `tsc` porque nesse ambiente o `node` não está no PATH, mesmo existindo `node_modules/.bin/tsc`. Então eu não validaria o projeto inteiro daqui ainda, mas a estrutura que eu mexeria primeiro é bem clara: ESLint + scripts + regra de camada Supabase + limpeza gradual de `any`.
